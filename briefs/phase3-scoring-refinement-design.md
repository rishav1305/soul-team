## 1. Percentile-Based Threshold Calibration

**Problem:** Current thresholds are educated guesses (e.g., "good" response time < 200ms). With no historical data, we can't know what's normal for our services.

**Approach:** After 2 weeks of data (~1,344 full collections):
1. Compute P25/P50/P75/P95 for each metric per product
2. Map percentiles to score bands:
   - P75+ → score 8-10 (healthy)
   - P25-P75 → score 5-7 (normal)
   - P10-P25 → score 3-4 (warning)
   - <P10 → score 1-2 (critical)
3. Store calibrated thresholds in `config.yaml` with `calibrated_at` timestamp
4. Re-calibrate monthly (or on-demand after major changes)

**Implementation:** Add `--calibrate` flag to collector.py that reads `health_scores` table, computes percentiles, and updates config.yaml.

## 2. Pipeline Phase Factor (Hawkeye Request)

**Problem:** Hawkeye correctly identified that pre-outreach pipelines will always show 0 interactions. Scoring this as "unhealthy" is misleading — it's a timing issue, not a quality issue.

**Design:**

Add a `phase` field to product config:

```yaml
products:
  scout:
    phase: "pre-outreach"  # Options: bootstrap, pre-outreach, active, mature
    phase_since: "2026-03-20"
    phase_until: "2026-03-27"  # Expected outreach activation
```

Phase-aware scoring adjustments:

| Phase | Service Weight | Usage Weight | Business Weight | Rationale |
|-------|---------------|-------------|-----------------|-----------|
| bootstrap | 0.70 | 0.20 | 0.10 | Service stability matters most early on |
| pre-outreach | 0.50 | 0.20 | 0.30 | Business metrics intentionally zero — don't penalize |
| active | 0.40 | 0.25 | 0.35 | Standard weights |
| mature | 0.30 | 0.25 | 0.45 | Business outcomes are what matter |

**Key rules:**
- In `pre-outreach` phase: if business_score components that require outreach (interactions, response_rate) are 0, substitute a neutral score (5.0) instead of 0
- Phase transitions can be manual (config edit) or automatic (e.g., bootstrap → active after first 100 API calls)
- Log phase transitions as alerts (severity=info) for audit trail

## 3. Alert Deduplication

**Problem:** Current system generates 26 alerts in 30 minutes for the same condition (all services down). This will flood inboxes.

**Design:**

Add `alert_state` table:

```sql
CREATE TABLE alert_state (
    product TEXT,
    metric TEXT,
    severity TEXT,
    first_fired_at TEXT,
    last_fired_at TEXT,
    fire_count INTEGER,
    acknowledged INTEGER DEFAULT 0,
    resolved_at TEXT,
    PRIMARY KEY (product, metric)
);
```

**Rules:**
1. **Cooldown:** Don't re-fire the same (product, metric, severity) alert within 4 hours
2. **Escalation:** If condition persists > 24 hours and unacknowledged, escalate severity by one level
3. **Resolution:** When metric recovers above threshold, fire a "resolved" alert and clear state
4. **Digest:** On each full collection, if >3 active alerts exist, write a single digest alert instead of individual ones

**Alert lifecycle:**
```
NEW → fire alert, write to inbox, create alert_state
REPEATED (within cooldown) → increment fire_count, don't write to inbox
ESCALATED (>24h unack) → fire escalated alert
RESOLVED → fire resolution alert, set resolved_at
```

## 4. Trend Detection Improvement

**Current:** Simple comparison with previous score (improving/stable/declining).

**Proposed:** Use last 6 data points (90 min window) with linear regression slope:
- slope > +0.3/hr → "improving"
- slope < -0.3/hr → "declining"
- else → "stable"

Also add a `volatility` flag: if std(last_6_scores) > 1.5, mark as "volatile" — indicates flapping service.
